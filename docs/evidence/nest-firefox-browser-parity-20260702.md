## Findings fixed during validation

1. Firefox BiDi could attach but navigation failed because Portage Firefox content processes crashed under the unavailable container user-namespace sandbox. The wrapper disables Firefox process sandboxes inside the pod while preserving the Kubernetes, NetworkPolicy, backend identity, sanitization, and final-purchase boundaries.
2. Firefox initially showed `SEC_ERROR_UNKNOWN_ISSUER` for `browser.eyrie` because `www-client/firefox` overwrote `libnssckbi.so` after the base cert class linked it to p11-kit trust. The tool class orders Firefox before `nest::base::certs`; the live image has `/usr/lib64/libnssckbi.so -> /usr/lib64/pkcs11/p11-kit-trust.so`.
3. The interim source-managed image used x11vnc plus noVNC and then client-side `resize=scale`, which made the Firefox surface appear enlarged compared with upstream KasmVNC. The current source replaces that shim with source-built KasmVNC from `kasmtech/KasmVNC`, built via `nest::lib::src_repo` and `nest::lib::build`, and serves the KasmVNC web client directly from `/opt/kasmweb/share/kasmvnc/www`.
4. The wrapper preserved runtime `HOME=/root`, putting Firefox state on ephemeral root storage instead of the mounted PVC. It now defaults `HOME` to `/home/kasm-user` unless `FIREFOX_HOME` is explicitly set; the live pod creates the profile under `/home/kasm-user/.mozilla/firefox/nest-secure-browser`.
5. Firefox could open smaller than the VNC framebuffer without a desktop window manager. The wrapper derives `FIREFOX_WIDTH`/`FIREFOX_HEIGHT` from `VNC_RESOLUTION`, passes them to Firefox, and uses `xdotool` to move/resize the visible Firefox window to `0,0 1365x768`. KasmVNC owns the browser-window/framebuffer behavior after that; there is no noVNC client-side scaling shim in the live path.
6. The Bitwarden enterprise policy was mounted at `/usr/lib/firefox/distribution/policies.json`, while Gentoo Firefox reads `/usr/lib64/firefox/distribution/policies.json`. The workload now mounts policy at the Gentoo path.
7. The image caches the public Bitwarden XPI at `/opt/nest/firefox/extensions/bitwarden.xpi` and exposes it through `/usr/lib64/firefox/distribution/extensions/{446900e4-71c2-419f-a6a7-df9c091e268b}.xpi`; the live profile has the extension XPI under `/home/kasm-user/.mozilla/firefox/nest-secure-browser/extensions/`.
8. The floating `latest` image tag did not by itself change the Deployment pod template, so Helm initially left the old pod running. `data/kubernetes/app/firefox.yaml` now carries a source-managed `image_revision` env value so KubeCM can perform a real rollout after this image rebuild.
9. KasmVNC accepted upward SetDesktopSize resize events, but the container had no desktop window manager to grow the Firefox top-level window after startup. The wrapper keeps a small `xdotool` loop running so Firefox follows the live KasmVNC framebuffer for both shrink and grow events, and the built KasmVNC web client forces `resize=remote` before its module starts so stale client-side `resize=scale` settings cannot reintroduce the enlargement path.
10. Joy rejected Openbox/window chrome and requested the existing Nest xmonad workstation path. The current image now starts from `nest/stage1/workstation:haswell`, removes Openbox from the tool class, installs a minimal fullscreen `/opt/nest/firefox/xmonad.hs`, and runs xmonad so Firefox extension popups keep transient focus without external titlebars/borders.
